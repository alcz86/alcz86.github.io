AI ROI Is a Query, Not an Anecdote

A measurement framework for a growing portfolio of AI automations: every workflow — agent, script, automation, or report — logs its outcomes through one importable logger into one central warehouse table. Cost, quality, volume, and ROI stop being claims in a slide deck and become rows you can GROUP BY.

01The problem — a portfolio nobody can see

Once you have half a dozen AI workflows in production — a triage agent, daily briefing automations, report generators, metadata enrichment jobs — a new problem appears: nobody knows what any of it is worth. No per-run cost, no quality trend, no failure visibility, no answer to "which of these should we scale and which should we kill?"

The uninstrumented default is anecdote-driven: automations feel useful, so they survive. That's exactly the kind of reasoning AI was supposed to replace. If AI adoption is worth leading, it's worth measuring like any other production system.

02Architecture — three tiers, one principle

The principle: one import, works everywhere. Instrumentation that takes more than two lines doesn't get adopted.

Tier 1 · Storage

One central table

A single warehouse table holds every run of every workflow — a deliberately flexible schema (typed core columns + a metadata JSON blob) so agents and report scripts coexist without migrations.

Tier 2 · Logging

One Python class

An importable WorkflowOutcomesLogger: declare the workflow's name, type, and environment once; call log() with the run's outcomes. Dual-writes to the warehouse and a local JSONL file, so an offline run never loses data.

Tier 3 · Analysis

Queries, dashboards, alerts

SQL templates and a portfolio dashboard over the one table; alert rules fire on cost spikes >20%, quality drops >10%, or failure rates >5%.

03The integration contract

This is the whole ask for a workflow owner — which is why adoption worked:

from outcomes_logger import WorkflowOutcomesLogger

logger = WorkflowOutcomesLogger(
    workflow_name="ticket-triage-agent",
    workflow_type="agent",        # or "script" / "automation" / "report"
    environment="prod",
)

# ... the workflow does its work ...

logger.log(
    status="success",
    items_output=18,              # e.g. tickets triaged
    cost_usd=0.05,
    quality_score=0.94,
    metadata={"action": "answered"},
)
logger.write_to_jsonl(Path("outcomes/outcomes.jsonl"))   # local fallback

05Design decisions worth stealing

  • Dual-write with a local JSONL fallback. The warehouse write can fail (auth, network, platform maintenance); the measurement must not. Local files reconcile later — instrumentation never becomes the reason a workflow fails.
  • Flexible schema over per-workflow tables. One table with a metadata blob beats a migration every time a new workflow type appears. Portfolio-level questions ("total spend this month, by type") are one query.
  • Alerts as part of the framework, not an afterthought. Cost and quality regressions in AI systems are silent by default — a model change or a prompt edit shifts behavior with no exception thrown. Thresholded alerts turn those into pages.
  • Compounding by design. Every newly instrumented workflow makes the portfolio view more complete — the framework gets more valuable with each adopter, which is what makes a measurement culture stick.

06Outcomes

  • The AI portfolio became a queryable production system: per-workflow cost, quality trend, and ROI on one dashboard, instead of scattered impressions.
  • "Should we scale this automation?" now has a numeric answer — and so does "should we kill it?".
  • The framework became the substrate for team-level AI-adoption reporting — the same rows roll up from a single run to a quarterly narrative.
