Cost-Aware Agent Operations: Same Model, Different Invoice

When an LLM vendor split billing — interactive sessions on a flat-rate subscription, headless/SDK calls on a metered credit pool — every automation's invocation path became an architecture decision. I turned that into an operating policy, a decision tree, an audit, and finally an executable rule the AI itself applies before building anything.

01The insight — billing path is an architecture decision

Mid-2026, our LLM vendor's billing changed: interactive CLI sessions kept drawing from the already-paid subscription, while headless/print-mode invocations and the agent SDK moved to a separately metered credit pool. Same model, same quality — different invoice.

The trap is subtle: an unattended cron job feels free because "it's just the same tool" — but the moment it runs headless, it quietly drains metered credits. Multiply by a team's worth of daily briefs, weekly reports, and always-on agents, and the difference between two functionally identical architectures is a real budget line. So the invocation path stopped being an implementation detail and became a reviewed decision.

02The decision tree

Every new automation passes through this gate before it's scaffolded:

1. Does it need to run unattended (cron / queue / trigger)?
   NO  → interactive session. Done.               ✅ FLAT-RATE
   YES ↓

2. Can it run as an INTERACTIVE session on a schedule?
   (loop-command + subagents · always-on host ·
    scheduled cloud session · CI runner launching interactive)
   YES → use one of those patterns.               ✅ FLAT-RATE  ← default
   NO  ↓

3. Is there a HARD constraint forcing headless/SDK?
   (must embed inside a service · sandboxed non-interactive env ·
    org-mandated production substrate for service agents)
   NO  → back to 2 — you don't need metered.
   YES ↓

4. Metered is justified — but never silently:
   ⚠ FLAG the credit cost before building at scale
   ⚠ CONFIRM with the budget owner
   ⚠ ROUTE through the central LLM gateway
   ⚠ BUDGET it deliberately (per-team caps)

03Four subscription-safe patterns

Pattern 1

Loop + subagents

One long-lived interactive session polls on a loop command and dispatches one subagent per work item. The workhorse pattern — it powers the always-on triage agent.

Pattern 2

Always-on interactive host

A scheduler (launchd) + terminal multiplexer + sleep-prevention keep the interactive session alive 24/7 — "always-on service" semantics without an SDK service.

Pattern 3

Scheduled cloud session

The vendor's scheduled-routine feature runs an interactive session in the cloud on cron — laptop-independent weekly reports without going headless.

Pattern 4

CI runner, interactive mode

A CI workflow step that launches the interactive CLI (not print mode) — team-hosted automation on shared runners, still on the flat rate.

04Not dogma — when metered is right

A cost policy that just says "never" gets routed around. Metered is the correct choice when the agent must live inside a running service (a subprocess CLI is the wrong architecture there), or when an automation graduates onto the org's mandated production substrate. In those cases the rule is flag → confirm → budget, not avoid — the failure mode being prevented is silent spend, not spend.

05Policy-as-code — the AI applies its own cost rules

The interesting part The policy isn't a wiki page; it's an agent skill — a machine-readable decision gate the AI assistant itself invokes whenever anyone asks it to build an automation, schedule a job, or make something run unattended. The skill outputs a verdict (FLAT-RATE or METERED), the decision-tree node that decided it, the concrete pattern to use, and — for metered — the flag-and-confirm ask. Governance that lives where the work happens can't be forgotten; the tool that would create the spend is the tool that enforces the policy.

The policy also shipped with an audit of existing automations: every cron, CI workflow, and scheduled routine was checked for headless invocations, and each hit either moved to a subscription-safe pattern or had its credit budget made explicit and intentional.

06Outcomes

  • Default automation path has zero marginal cost and — structurally — no runaway-spend failure mode.
  • Every metered exception is flagged, confirmed, and budgeted before it exists; silent credit drain eliminated by construction.
  • The one-liner that stuck in reviews: "Default to interactive loop + subagents on the flat rate; go headless only when a hard constraint forces it — and flag the cost first."
