01The problem

Support tickets landing in a data-team queue each waited hours for an analyst to pick them up, re-trace the same lineage and metric questions by hand, and reply — and a meaningful share turned out to be already-answered or mis-scoped. The goal: give every ticket a correct, sourced first response in minutes, without granting an AI system any write access to production data.

The second, more interesting problem appeared after launch: an always-on agent is an operations liability. It can die, hang on an interactive prompt, or keep looping while silently doing nothing. Babysitting it would erase the time it saved. So reliability — and eventually self-remediation — became first-class features, not afterthoughts.

02Architecture — judgment in the LLM, guarantees in code

The design principle throughout: the LLM does the open-ended reasoning; everything that must be correct every time is deterministic, unit-tested Python with no model in the path.

LLM layer

Two agent skills

A thin scanner (detect → dedupe → dispatch, polled every 5 minutes) and a brain (classify → investigate → draft → post), one sub-agent per ticket. Prompts encode an investigation discipline, not answers.

Deterministic core

Pure-stdlib Python modules

Config, normalization, classification gates, dedupe/state, heartbeat, incident log, remediation, PR authoring — 62 tests, no network or clock in test paths (both injected).

04The full loop — the agent repairs its own operation

The most common recurring failure: the agent invokes a command that isn't in its unattended-permission allowlist, hits an approval prompt, and freezes — every run, until a human notices, diagnoses it, writes the allowlist rule, and applies it. That whole chain is now automated except the one step that should stay human: approving a security change.

The interesting engineering is in step 3 — a watchdog that writes code into a repo a live agent is running from has to be paranoid:

  • Isolated git worktree. The PR is authored in a throwaway worktree, never in the live tree the agent loops in — an in-flight scan is untouched.
  • Deterministic branch names. The branch is a hash of the rule, not a timestamp — so a repeat detection force-pushes the same branch and reuses the existing PR instead of opening duplicates. Idempotency by construction.
  • Injected runner, per-call timeouts. Every git/gh call goes through an injectable subprocess wrapper with a 60s timeout — the watchdog can never hang on a network stall, and the whole flow is testable without touching git.
  • Graceful degradation. Any failure (no gh, offline, auth expired) returns ok=False and the system falls back to its previous behavior — a chat message with the manual one-liner. Adding the new capability could not make the old one worse.
  • The gitignored-config problem. The live allowlist lives in a machine-local, gitignored settings file a PR can't touch. Solution: the PR edits a tracked ledger file instead, and a sync script — run by the nightly restart after it pulls — copies merged ledger rules into the local settings. The audit trail lives in git; the secret-adjacent file stays local.
  • Propose, never auto-apply. An allowlist rule widens what the agent may run unattended — a permission change. Those stay human-gated by design; a reviewable diff with merge-as-approval beats both silent auto-writes and "please run this script" pings.

05Engineering practices worth stealing

TDD with injected effects

Clock, subprocess runner, and notifier are all parameters. 62 tests cover the deterministic core — including the PR-authoring flow, tested with a fake runner and zero real git.

Idempotency everywhere

Comment markers, state files, proposal dedup, hash-derived branch names. Every action is safe to repeat — mandatory when a scheduler and an agent share a system.

Incidents as data

Failures aren't just alerts — they're structured, signed records. Recurrence detection, dedup, and remediation are all simple queries over that log.

Cost-aware runtime

The agent runs as an interactive flat-rate session rather than metered API/headless SDK calls — zero marginal cost per ticket and no runaway-spend failure mode.

06Outcomes

  • First response on data tickets: hours → minutes, with every answer citing the doc and catalog entry behind it — including catching requests for work that was already done.
  • Silent-failure MTTR: unbounded → ≤15 minutes, with exactly one page per incident.
  • Recurring operational failures: investigate-and-patch by hand → review a bot-authored PR.
  • Marginal cost per ticket: $0; blast radius bounded at "a ticket comment" by construction.

07CV-ready summary

  • Designed and built an always-on LLM ticket-triage agent (Claude Code + Python) that investigates data-support tickets read-only against docs, a metadata catalog, and BigQuery, and posts sourced advisory answers — cutting first response from hours to minutes at zero marginal cost.
  • Engineered its reliability layer: a launchd watchdog detecting three silent-failure modes (down / stuck / stale-heartbeat) with edge-triggered alerting and nightly last-known-good restarts — silent-failure MTTR bounded at 15 minutes.
  • Closed the self-remediation loop: on recurring failures the system diagnoses the root cause and opens its own fix pull request — authored in an isolated git worktree with hash-deterministic branches (idempotent), per-call timeouts, and graceful fallback — keeping the human in the loop only as PR reviewer for permission changes.
  • Built the deterministic core test-first (62 unit tests; injected clocks, runners, notifiers) with idempotency guarantees across every side effect.
