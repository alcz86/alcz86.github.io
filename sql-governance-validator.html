Governance-as-Code: Validating SQL Against the Metadata Catalog

A web app where anyone pastes a SQL query — or the URL of a running data app — and gets two machine-checked verdicts: are the source tables governed? and do the metric formulas match their certified definitions? It turns "is this number trustworthy?" from a half-day manual investigation into a one-paste check.

01The problem — the gold star that lies politely

Every data org trusts its "golden datasets": certified tables and metrics that someone, at some point, declared correct. Keyword: at some point. Schemas change, metrics get redefined, logic quietly moves — and the certification badge keeps glowing. A query written six months ago keeps aggregating yesterday's formula over today's data, with full confidence.

Checking whether a given query is actually aligned with the catalog meant a manual investigation: trace each source table's lineage, find the certified metric definition, compare formulas by eye. Nobody does that for every dashboard — so drift accumulates silently. The fix isn't more certification; it's making the check cheap enough to run every time.

03The hard part — one table name, many physical tables

In a real data platform a logical table name resolves to many physical copies: warehouse views, ETL staging layers, per-project clones. The worst case observed: one name → 18 fully-qualified names across platforms, where exactly one copy was governed.

Resolution strategy The validator ranks all copies in a single catalog search call — data-product membership rides along in the search response, so no per-copy lookups are needed. It prefers the governed copy by default, pins the exact copy when the query names a schema qualifier, and — the genuinely useful move — when the query reads an ungoverned copy that has a governed equivalent, it tells the author which fully-qualified name to read from instead. The verdict isn't just "bad"; it's "here's the good one."

04Deterministic by design — the LLM only where it belongs

  • The SQL path has no model in it. Parsing is a sqlglot AST walk; formula comparison is normalization + equality. Same input, same verdict, every time — a governance tool that hallucinated governance would be worse than none.
  • The prose path is where the LLM earns its keep. Paste analytics text with a claimed number, and an LLM extracts the claim — then the deterministic engine recomputes it from the certified source and reconciles the values. Judgment for extraction, code for the verdict.
  • URL mode closes the loop for deployed apps: given a running data app's URL, the validator fetches the app's SQL and runs both stages on what's actually deployed — not what the author remembers deploying.

05Architecture

A Python data app (FastAPI backend + HTML UI) hosted on the team's cloud data platform, git-backed. The metadata catalog (OpenMetadata) is the single source of truth for both stages — table governance and certified formulas come from its REST API, never from a hardcoded list. The app carries no state of its own: catalog changes are reflected on the next paste.

06Outcomes

  • Source-trust checking went from a manual lineage investigation to a one-paste, seconds-long check — cheap enough that it actually happens.
  • Caught real metric drift in live queries — including a query aggregating a plausible-but-wrong variant of a certified metric that eyeballing had passed for months.
  • Verdicts are reproducible and explainable: every one links to the catalog entry that decided it.
